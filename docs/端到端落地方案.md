# AI 语音聊天 RTC 优化：端到端落地方案

> **日期**：2026-03-12
>
> **背景**：本项目已完成 Opus 编解码层面的 FEC 分析和优化研究（LBRR/DRED/新方案探索）。本文档解决下一步问题：**如何将 Opus 层优化成果集成到真实的 AI 语音聊天 RTC 链路中，端到端打通并量化呈现优化效果**。

---

## 1. AI 语音聊天的完整链路

```
用户端（客户端）                              服务端
┌──────────────────┐                    ┌──────────────────────────┐
│ 麦克风采集        │                    │ SFU / 媒体服务器          │
│   ↓              │                    │   ↓                      │
│ 音频前处理        │   RTP/SRTP         │ Opus 解码                │
│ (AEC/NS/AGC)     │ ──────────────→    │   ↓                      │
│   ↓              │    弱网信道         │ STT（语音识别）           │
│ Opus 编码        │ ←──────────────    │   ↓                      │
│ (+LBRR/DRED)     │   RTP/SRTP         │ LLM（大语言模型）         │
│   ↓              │                    │   ↓                      │
│ RTP 打包/发送     │                    │ TTS（语音合成）           │
│                  │                    │   ↓                      │
│ ← 接收/解包      │                    │ Opus 编码 (+LBRR/DRED)   │
│ Opus 解码        │                    │   ↓                      │
│ (+DRED恢复/PLC)  │                    │ RTP 打包/发送             │
│   ↓              │                    └──────────────────────────┘
│ 扬声器播放       │
└──────────────────┘
```

### Opus 优化影响的关键环节

| 链路环节 | Opus 优化的影响 | 可观测指标 |
|---------|----------------|-----------|
| **用户→服务端** 上行语音 | DRED/LBRR 保护用户语音不因丢包劣化 | STT 识别准确率、丢包恢复率 |
| **服务端→用户** 下行语音 | DRED/LBRR 保护 AI 回复语音 | 播放端音质 MOS/ViSQOL、卡顿率 |
| **端到端延迟** | VBR+DRED 的码率波动影响排队延迟 | 口到耳延迟（mouth-to-ear） |
| **弱网体验** | 丢包/抖动/突发场景下语音连续性 | 可懂度、自然度主观评分 |

## 2. 优化效果的呈现维度

### 2.1 必须呈现的核心指标

| 指标 | 含义 | 测量方法 | 目标 |
|------|------|---------|------|
| **丢包恢复率** | 丢失的包中被 FEC 恢复的比例 | opus_sim 统计 | DRED VBR ≥ 95% |
| **音频质量 MOS** | 主观听感评分（1-5 分） | ViSQOL 客观估计 | 弱网下 ≥ 3.5 |
| **端到端延迟** | 麦克风到扬声器的总延迟 | RTP 时间戳差 + 播放缓冲 | < 500ms（含 LLM） |
| **卡顿率** | 音频播放中断次数/总时长 | 解码端统计连续 PLC 帧 | < 2% |
| **码率开销** | FEC 冗余导致的额外带宽 | 编码后包大小统计 | VBR < 40kbps |

均匀丢包 20%        ✓         ✓         ✓           ✓

## 3. 端到端打通方案

### 3.1 方案概览：三层渐进验证

```
第一层：离线仿真验证（已有，本项目 opus_sim）
  → 验证 Opus 编解码层的 FEC 效果
  → 输出：恢复率、SNR、音频文件

第二层：UDP 实时回环验证（已有，sender/receiver）
  → 验证实时传输 + 网络损伤下的效果
  → 输出：实时恢复率、端到端延迟

第三层：WebRTC 全链路验证（需新建）
  → 验证完整 RTC 栈中的效果
  → 输出：真实场景 MOS、与 AI 交互的端到端体验
```

### 3.2 第三层：WebRTC 全链路方案（推荐）

#### 方案 A：基于 LiveKit + 自定义 Opus（推荐）

```
架构:

  浏览器/客户端                LiveKit SFU              AI 后端
  ┌───────────┐            ┌──────────────┐        ┌──────────┐
  │ WebRTC    │ ──SRTP──→  │ 转发/录制     │ ─────→ │ STT      │
  │ (Opus)    │ ←──SRTP──  │              │ ←───── │ LLM      │
  └───────────┘            │ 可插入自定义   │        │ TTS      │
                           │ Opus 编解码   │        │ (Opus编码)│
                           └──────────────┘        └──────────┘
```

**选择 LiveKit 的理由**：
- 原生 Go 实现，内置 Pion WebRTC，可直接替换 Opus 库
- 官方 Agents 框架专为 AI 语音场景设计（STT/LLM/TTS 集成）
- 开源，单节点部署无外部依赖
- 内置房间录制功能，方便离线音质分析

**集成步骤**：

1. **替换 Opus 库**：LiveKit 使用 Pion 的 Opus 绑定（cgo 调用 libopus）。将 `opus-install/` 的 1.6.1 库替换进去，启用 DRED
2. **修改编码参数**：在 LiveKit 的音频发布逻辑中设置 `OPUS_SET_INBAND_FEC(1)`、`OPUS_SET_DRED_DURATION(n)`、`OPUS_SET_PACKET_LOSS_PERC(plp)`
3. **修改解码逻辑**：在音频订阅逻辑中加入 DRED 解析和恢复
4. **AI 后端对接**：用 LiveKit Agents 框架连接 STT（如 Whisper）+ LLM + TTS（如 Coqui/XTTS）

#### 方案 B：基于 Pion WebRTC 的轻量 Demo

```
架构（更简单，适合快速验证）:

  客户端 A（模拟用户）              客户端 B（模拟 AI）
  ┌───────────────┐              ┌───────────────┐
  │ 读取 WAV       │              │ 接收 RTP       │
  │ Opus 编码      │  Pion P2P    │ Opus 解码      │
  │ (+DRED/LBRR)  │ ──────────→  │ (+DRED/PLC)    │
  │ RTP 发送       │  tc netem    │ 写入 WAV       │
  └───────────────┘   网络损伤    └───────────────┘
```

**选择 Pion 的理由**：
- 纯 Go 实现，代码量小，易于修改
- `pion/opus` 库可直接链接自定义 libopus
- 不需要媒体服务器，P2P 直连

**集成步骤**：

1. 用 `pion/webrtc` 建立 P2P 连接
2. 发送端读取 WAV 文件，用自定义 Opus 1.6.1 编码（启用 DRED）
3. 在网络路径上用 `tc netem` 注入丢包/延迟/抖动
4. 接收端解码并写入 WAV，同时记录恢复统计
5. 用 ViSQOL 对比原始和解码后音频

#### 方案 C：基于现有 sender/receiver 扩展（最快落地）

```
架构（在本项目基础上扩展）:

  opus_sender                    opus_receiver
  ┌──────────────┐              ┌──────────────────┐
  │ WAV → Opus   │  UDP + tc    │ UDP → Opus 解码   │
  │ +DRED/LBRR   │ ──────────→  │ +DRED/PLC 恢复    │
  │ RTP 发送      │   netem     │ → WAV 输出         │
  │              │              │ → 统计报告          │
  └──────────────┘              └──────────────────┘
                                        ↓
                                  ViSQOL 音质评估
                                  Python 可视化报告
```

**优点**：基于已有代码，最快可用
**缺点**：不走 WebRTC 栈（无 ICE/DTLS/RTCP），不能直接证明"WebRTC 中也有效"

**核心代码思路**（`sender/main.go` 关键部分）：

```go
// 创建 PeerConnection
pc, _ := webrtc.NewPeerConnection(config)

// 添加 Opus 音轨
track, _ := webrtc.NewTrackLocalStaticSample(
    webrtc.RTPCodecCapability{MimeType: "audio/opus"},
    "audio", "voice",
)
pc.AddTrack(track)

// 读取 WAV，用自定义 libopus (含 DRED) 编码
encoder := opus.NewEncoder(48000, 1, opus.AppVoIP)
encoder.SetInBandFEC(true)
encoder.SetPacketLossPerc(10)
encoder.SetDREDDuration(5)  // 需要 opus 绑定支持

// 循环编码并发送
for frame := range audioFrames {
    encoded := encoder.Encode(frame)
    track.WriteSample(media.Sample{Data: encoded, Duration: 20*time.Millisecond})
}
```

## 8. 关键风险和注意事项

### 8.1 Opus 版本兼容性

当前 WebRTC 生态（Chrome、Pion、LiveKit）内置的 Opus 通常是 1.3-1.4 版本，**不含 DRED 支持**。集成 1.6.1 需要：

- 替换链接的 libopus 动态库
- 确保 DRED API（`opus_dred_parse`、`opus_decoder_dred_decode`）可用
- DNN 权重文件（`weights_blob.bin`）需跟随部署

### 8.2 CBR vs VBR

本项目实验已证明：**1.6.1 的 DRED 在 CBR 下对复杂语音恢复率仅 13-34%，VBR 下达 100%**。
WebRTC 默认使用 VBR，这是有利的。但需注意：
- VBR 的码率波动可能影响拥塞控制（GCC/BBR）
- 突然增大的包可能被中间设备丢弃

### 8.3 DRED 延迟影响

DRED 本身不增加算法延迟（冗余嵌入当前包），但：
- DNN 编码端推理增加约 1-2ms 计算延迟
- VBR 的大包可能增加传输延迟
- DRED 恢复需要等待下一个到达的包（1 帧延迟）

### 8.4 STT 对音质的敏感度

AI 语音聊天中，STT 是上行语音质量的"消费者"。需要测试：
- 不同丢包率 + 不同 FEC 方案 → STT 识别准确率（WER/CER）
- DRED 恢复的音频 vs PLC 音频 → 哪个对 STT 更友好
- 这是一个独立于 MOS 的重要维度
